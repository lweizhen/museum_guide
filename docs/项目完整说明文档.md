## 1. 项目一句话说明

本项目是一个面向博物馆文物导览场景的中文语音导览问答系统。用户可以上传文物图片并输入问题，系统会先识别候选文物，再结合文物知识库生成导览式回答，最后可以把回答转换为语音播放。

项目同时支持两类主要方案：

| 方案 | 名称 | 核心思路 | 适用场景 |
|---|---|---|---|
| 方案 A | 图像检索增强文本问答 | 图片先经过图像检索识别候选文物，再进入文本 RAG 问答链路 | 工程稳定、可解释性强、演示可靠 |
| 方案 B4 | Qwen2.5-VL + RAG + LoRA | 图片、问题和检索上下文共同输入多模态大模型，并加载 LoRA adapter | 多模态模型参与图文联合推理，适合展示模型增强方案 |

当前网页展示主要提供两个入口：

| 网页入口 | 后端执行位置 | 说明 |
|---|---|---|
| 方案 A | 远程 GPU 服务 | 虽然方案 A 本身不是端到端多模态大模型，但为了统一演示环境，也通过远程服务完成图片检索、文本检索和文本生成 |
| 方案 B4 | 远程 GPU 服务 | 调用远程 Qwen2.5-VL + LoRA 推理服务，完成检索增强多模态回答 |

## 2. 项目解决的问题

传统博物馆导览方式主要依赖展签、二维码、人工讲解或固定音频。它们的问题是：

| 问题 | 表现 |
|---|---|
| 交互不足 | 观众只能被动阅读或收听，无法自由提问 |
| 识别门槛 | 观众看到文物时不一定知道名称，难以直接搜索 |
| 回答不稳定 | 直接使用大模型容易编造年代、馆藏、用途等事实 |
| 展示方式单一 | 只有文字或固定音频，缺少个性化导览体验 |

因此本项目设计了一个“图片识别 + 知识检索 + 大模型回答 + 语音播报”的系统。核心目标不是让模型自由聊天，而是让回答尽量基于知识库、可追踪、可评测、可演示。

## 3. 核心技术概念

### 3.1 RAG

RAG 是 Retrieval-Augmented Generation 的缩写，中文通常称为检索增强生成。它的基本思想是：不要只依赖大模型自身记忆，而是在生成回答前先从知识库中检索相关资料，再把这些资料放进提示词中，让模型依据资料回答。

本项目中，RAG 主要用于：

| 位置 | 作用 |
|---|---|
| 方案 A | 图片检索得到候选文物后，再用文本索引召回对应知识块，最后让文本大模型回答 |
| 方案 B4 | 先检索文物知识上下文，再把图片、问题和知识上下文一起输入 Qwen2.5-VL |

### 3.2 FAISS

FAISS 是一个高效向量检索库。项目中有两个 FAISS 索引：

| 索引 | 输入 | 输出 | 文件 |
|---|---|---|---|
| 文本索引 | 用户问题或文物名称 | 相似文物知识块 | `index/exhibits.index` |
| 图片索引 | 上传图片 | 相似文物图片和元数据 | `index/exhibits_images.index` |

### 3.3 Qwen2.5-VL

Qwen2.5-VL 是多模态大模型，可以同时接收图片和文本。项目中它用于：

| 链路 | 作用 |
|---|---|
| `vl_direct` | 直接根据图片和问题回答，不加入知识库 |
| `vl_rag` | 根据图片、问题和检索上下文回答 |
| `vl_lora` | 加载 LoRA 后直接图文问答 |
| `vl_rag_lora` | 加载 LoRA 后结合检索上下文回答 |

### 3.4 LoRA

LoRA 是 Low-Rank Adaptation 的缩写，是一种参数高效微调方法。它不会修改大模型所有参数，而是在部分线性层中加入低秩可训练矩阵，只训练这些 adapter 参数。

本项目使用 LoRA 的目的不是让模型记住所有知识，而是让模型更适应：

| 目标 | 说明 |
|---|---|
| 闭集文物识别 | 更熟悉当前数据集中的文物名称和视觉特征 |
| 导览式表达 | 回答更像中文博物馆导览员，而不是机械字段拼接 |
| 带知识上下文问答 | 学会优先依据检索资料回答 |

### 3.5 TTS

TTS 是 Text-to-Speech 的缩写，即文本转语音。本项目使用 `edge-tts` 把生成的中文回答转换成 MP3，在 Streamlit 页面中播放。

## 4. 总体架构

项目可以按五层理解：

| 层级 | 内容 | 对应目录或文件 |
|---|---|---|
| 数据层 | 文物文本资料、文物图片、多模态训练与测试数据 | `data/` |
| 索引层 | 文本向量索引、图片向量索引 | `index/`、`src/retriever.py`、`src/image_retriever.py` |
| 模型层 | 文本 LLM、Qwen2.5-VL、LoRA、TTS | `src/llm.py`、`src/hf_qwen_vl.py`、`src/tts.py` |
| 应用层 | Streamlit 网页、命令行、远程 GPU API | `app.py`、`run_cli.py`、`scripts/serve/` |
| 评测层 | 五链路评测、文本指标、质量裁判 | `scripts/eval/`、`scripts/judge/`、`outputs/` |

系统的主流程可以理解为：

```text
用户上传图片和问题
        |
        v
图片编码与图像检索
        |
        v
获得候选文物名称、相似度、馆藏等元数据
        |
        v
根据候选文物和问题检索文本知识库
        |
        v
构造提示词
        |
        +-----------------------------+
        |                             |
        v                             v
方案 A：文本 RAG 生成回答       方案 B4：Qwen2.5-VL+RAG+LoRA 生成回答
        |                             |
        +-------------+---------------+
                      |
                      v
              网页展示回答和依据
                      |
                      v
                TTS 语音播报
```

## 5. 两条主要演示方案

### 5.1 方案 A：图像检索增强文本问答

方案 A 的关键点是“拆分”。它不让一个多模态大模型直接完成所有事情，而是把任务拆成几个环节：

```text
图片 -> 图像检索 -> 候选文物 -> 文本检索 -> 文本 RAG -> 回答
```

优点：

| 优点 | 说明 |
|---|---|
| 可解释 | 可以看到候选文物、相似度和检索上下文 |
| 稳定 | 事实回答主要依赖知识库 |
| 易排错 | 如果错了，可以判断是图片检索错、文本检索错，还是生成模型错 |

缺点：

| 缺点 | 说明 |
|---|---|
| 视觉理解弱一些 | 图片只用于检索，不直接参与回答生成 |
| 依赖索引质量 | 如果图像检索阶段候选错误，后续回答容易跟着偏 |

### 5.2 方案 B4：Qwen2.5-VL + RAG + LoRA

方案 B4 是当前多模态增强方案的主展示链路：

```text
图片 -> 图像检索 -> 文本检索 -> 图片 + 问题 + 知识上下文 -> Qwen2.5-VL + LoRA -> 回答
```

它和方案 A 的主要区别是：

| 对比点 | 方案 A | 方案 B4 |
|---|---|---|
| 图片是否进入大模型 | 否 | 是 |
| 生成模型 | 文本 LLM | 多模态 Qwen2.5-VL |
| 是否加载 LoRA | 否 | 是 |
| 研究意义 | 检索主导、生成跟随 | 多模态模型主导、外部知识辅助 |

方案 B4 的优势是：模型既能看到图片，又能看到检索出来的文物资料，还能通过 LoRA 适配导览任务。

## 6. 统一五链路评测设计

为了做论文实验和消融分析，项目定义了五条链路：

| 链路名称 | 中文说明 | 图片是否进入 VLM | 是否使用 RAG | 是否使用 LoRA |
|---|---|---:|---:|---:|
| `retrieval_rag_text` | 图像检索增强文本问答 | 否 | 是 | 否 |
| `vl_direct` | Qwen2.5-VL 直接问答 | 是 | 否 | 否 |
| `vl_rag` | Qwen2.5-VL + RAG | 是 | 是 | 否 |
| `vl_lora` | Qwen2.5-VL + LoRA | 是 | 否 | 是 |
| `vl_rag_lora` | Qwen2.5-VL + RAG + LoRA | 是 | 是 | 是 |

这五条链路对应论文中的主要对比实验：

| 实验问题 | 对应比较 |
|---|---|
| 多模态大模型直接问答是否可靠 | `vl_direct` |
| 检索增强是否有效 | `vl_direct` vs `vl_rag` |
| LoRA 是否有效 | `vl_direct` vs `vl_lora` |
| RAG 和 LoRA 是否互补 | `vl_rag`、`vl_lora`、`vl_rag_lora` |
| 工程基线是否稳定 | `retrieval_rag_text` vs 其他链路 |

## 8. `src/` 核心代码说明

`src/` 是项目的核心业务代码目录。网页、命令行、评测脚本和远程服务都会调用这里的模块。

### 8.1 `src/config.py`

作用：项目配置中心。

它负责读取：

| 配置类型 | 示例 |
|---|---|
| LLM provider | `ollama` 或 `dashscope` |
| Ollama 地址和模型 | `http://localhost:11434`、`qwen2.5:7b` |
| DashScope 模型 | `qwen3.5-plus`、`qwen-vl-plus` |
| 远程 GPU 服务地址 | `http://127.0.0.1:8000` |
| 文本知识库路径 | `data/exhibits_combined.txt` |
| 文本索引路径 | `index/exhibits.index` |
| 图片索引配置 | 图片 CSV、缓存目录、FAISS 索引、元数据 |
| TTS 配置 | 语音角色、输出目录 |

核心函数：

| 函数 | 作用 |
|---|---|
| `_load_yaml_config()` | 读取项目根目录的 `config.yaml` |
| `_get()` | 按“配置文件 > 环境变量 > 默认值”的顺序获取配置 |
| `get_api_key()` | 获取 DashScope API Key |

配置优先级：

```text
config.yaml > 环境变量 > src/config.py 中的默认值
```

### 8.2 `src/kb.py`

作用：读取文物知识库。

知识库文件默认是：

```text
data/exhibits_combined.txt
```

这个文件中每个文物知识块之间用空行分隔。`kb.py` 会把它读取成一个列表，供文本检索模块使用。

核心函数：

| 函数 | 作用 |
|---|---|
| `load_docs()` | 读取知识库文件，并按空行切分成多个文物知识块 |

### 8.3 `src/embedder.py`

作用：文本向量编码。

文本检索时，系统需要把用户问题和文物知识块都转换为向量，再用 FAISS 检索相似内容。这个模块封装了文本 embedding 模型。

默认模型：

```text
paraphrase-multilingual-MiniLM-L12-v2
```

核心函数：

| 函数 | 作用 |
|---|---|
| `_resolve_local_model_path()` | 优先查找本地缓存模型，避免远程环境无法访问 Hugging Face 时失败 |
| `get_embed_model()` | 加载并缓存文本向量模型 |
| `encode_texts()` | 把文本列表编码为向量 |

### 8.4 `src/retriever.py`

作用：文本知识库检索。

它负责把用户问题编码成向量，在 `index/exhibits.index` 中查找最相关的文物知识块。

核心函数：

| 函数 | 作用 |
|---|---|
| `get_index()` | 加载文本 FAISS 索引 |
| `get_docs()` | 加载知识库文本块 |
| `_extract_name()` | 从知识块中提取文物名称 |
| `_extract_aliases()` | 从知识块中提取文物别名或相关名称 |
| `_expand_query_with_doc_aliases()` | 根据知识块中的名称扩展查询，提升召回稳定性 |
| `retrieve()` | 执行文本检索，返回候选知识块和相似度 |

典型调用链：

```text
用户问题
  -> encode_texts()
  -> FAISS search
  -> 返回 Top-K 文物知识块
```

### 8.5 `src/image_embedder.py`

作用：图片和图文检索向量编码。

图像检索需要把图片转换成向量。本项目使用 SentenceTransformers 中的 CLIP 模型来做图片向量和文本向量编码。

默认模型：

```text
clip-ViT-B-32
```

核心函数：

| 函数 | 作用 |
|---|---|
| `_resolve_local_model_path()` | 优先解析本地图片向量模型路径 |
| `get_image_model()` | 加载并缓存 CLIP 图像模型 |
| `_normalize_embeddings()` | 对向量做归一化，方便用内积近似余弦相似度 |
| `encode_images()` | 批量编码图片 |
| `encode_image()` | 编码单张图片 |
| `encode_texts()` | 用同一 CLIP 模型编码文本 |
| `encode_text()` | 编码单条文本 |

### 8.6 `src/image_index.py`

作用：构建图片索引时的数据准备工具。

它负责从 CSV 中读取文物图片链接，下载图片，检查图片是否有效，并保存图片元数据。

核心类和函数：

| 名称 | 作用 |
|---|---|
| `ImageRecord` | 图片记录数据类，保存文物名称、时代、馆藏、图片 URL、图片路径等 |
| `load_image_rows()` | 读取图片 CSV |
| `parse_image_urls()` | 解析一条记录里的多个图片链接 |
| `build_image_records()` | 将 CSV 行转换为图片记录对象 |
| `download_image()` | 下载远程图片到本地缓存 |
| `open_local_image()` | 打开本地图片 |
| `is_valid_local_image()` | 检查本地图片是否可读取 |
| `dump_image_meta()` | 保存图片元数据 JSON |

### 8.7 `src/image_retriever.py`

作用：图片检索。

当用户上传图片后，这个模块会：

1. 把图片编码为向量。
2. 加载图片 FAISS 索引。
3. 搜索相似图片。
4. 返回候选文物名称、时代、馆藏、相似度等信息。
5. 判断候选结果是否足够可信。

核心函数：

| 函数 | 作用 |
|---|---|
| `get_image_index()` | 加载图片 FAISS 索引 |
| `get_image_meta()` | 加载图片元数据 |
| `search_image()` | 输入图片，返回 Top-K 相似文物图片 |
| `assess_image_match_confidence()` | 根据相似度和候选差距判断图片检索是否可信 |

### 8.8 `src/prompt.py`

作用：提示词构造。

这个模块非常关键，它决定了模型看到什么上下文、以什么身份回答、回答时遵守什么约束。

主要提示词类型：

| 函数 | 用途 |
|---|---|
| `build_prompt()` | 文本 RAG 链路的提示词 |
| `build_multimodal_direct_prompt()` | 多模态 direct 链路提示词 |
| `build_multimodal_grounded_prompt()` | 多模态 RAG / LoRA 链路提示词 |
| `build_multimodal_guide_prompt()` | 导览式回答提示词 |
| `build_citation()` | 根据检索上下文生成资料来源说明 |

当前 `build_multimodal_grounded_prompt()` 会根据问题类型自动切换：

| 问题类型 | 使用策略 |
|---|---|
| 字段型问题 | 更强调名称、年代、馆藏、用途等事实准确 |
| 导览型问题 | 更强调讲解口吻、重点突出和观众友好表达 |

### 8.9 `src/llm.py`

作用：统一封装大模型调用。

它支持多种模型后端：

| 后端 | 说明 |
|---|---|
| Ollama | 本地或远程部署的开源模型服务 |
| DashScope | 阿里云 DashScope API |
| OpenAI-compatible | 兼容 OpenAI 接口格式的模型服务 |

核心函数：

| 函数 | 作用 |
|---|---|
| `call_llm()` | 调用文本大模型生成回答 |
| `call_judge_llm()` | 调用裁判模型，用于评价答案质量 |
| `call_multimodal_llm()` | 调用多模态模型，输入图片和文本 |
| `_call_ollama()` | 调用 Ollama 文本模型 |
| `_call_ollama_multimodal()` | 调用 Ollama 多模态模型 |
| `_call_dashscope()` | 调用 DashScope 文本模型 |
| `_call_dashscope_multimodal()` | 调用 DashScope 多模态模型 |

### 8.10 `src/hf_qwen_vl.py`

作用：Hugging Face 版 Qwen2.5-VL 推理封装。

远程 GPU 服务和统一五链路评测会用到它。它负责加载：

| 内容 | 说明 |
|---|---|
| Qwen2.5-VL 基础模型 | 例如 `/root/autodl-tmp/models/Qwen2.5-VL-3B-Instruct` |
| LoRA adapter | 例如 `outputs/lora/qwen25vl3b_qkvo_llm_v2` |
| Processor | 处理图片和文本输入 |

核心类：

| 类 | 作用 |
|---|---|
| `HfQwenVlGenerator` | 封装模型加载、图片处理、文本生成 |

常用方法：

| 方法 | 作用 |
|---|---|
| `generate()` | 输入图片路径和提示词，返回模型回答 |

### 8.11 `src/remote_vl.py`

作用：远程 GPU 导览服务客户端。

本地 Streamlit 网页不直接加载大模型，而是把图片和问题通过 HTTP 请求发送到远程 GPU 服务。

| 函数 | 作用 |
|---|---|
| `_post_image_request()` | 把图片和问题上传给远程服务 |

| `call_remote_vl_rag_lora()` | 调用远程 B4 接口 |

网页中的两个方案按钮最终都会调用这个模块。

### 8.12 `src/tts.py`

作用：文本转语音。

生成回答后，系统会调用这个模块把中文回答转成 MP3。

核心函数：

| 函数 | 作用 |
|---|---|
| `_gen_mp3()` | 异步调用 edge-tts 生成 MP3 |
| `text_to_mp3()` | 对外提供同步接口，返回音频文件路径 |

输出目录默认是：

```text
outputs/media/
```

### 8.13 `src/eval_utils.py`

作用：评测工具函数。

多个评测脚本会共用这里的文本清洗、字段提取、答案打分和 JSON 解析函数。

核心函数：

| 函数 | 作用 |
|---|---|
| `read_jsonl()` | 读取 JSONL 文件 |
| `normalize_text()` | 文本归一化 |
| `strip_model_artifacts()` | 去掉模型输出中的异常标记 |
| `extract_field()` | 从知识块中提取指定字段 |
| `build_text_query()` | 构造检索查询 |
| `safe_div()` | 安全除法，避免除零 |
| `score_answer()` | 对答案进行自动打分 |
| `contains_expected()` | 判断回答是否包含参考答案 |
| `parse_json_object()` | 从模型输出中解析 JSON 对象 |

### 8.14 `src/progress.py`

长时间评测、批处理或训练数据构建时，用它显示进度、耗时和预计剩余时间。

## 11. `scripts/finetune/` LoRA 微调脚本

### 11.1 `scripts/finetune/prepare_qwen25vl_lora_data.py`

作用：把 `train_lora.jsonl` 等数据转换成 Qwen2.5-VL 微调需要的消息格式。

运行：

```bash
python scripts/finetune/prepare_qwen25vl_lora_data.py
```

### 11.2 `scripts/finetune/train_qwen25vl_lora.py`

作用：训练 Qwen2.5-VL 的 LoRA adapter。

基础运行：

```bash
python scripts/finetune/train_qwen25vl_lora.py \
  --model-path /root/autodl-tmp/models/Qwen2.5-VL-3B-Instruct \
  --output-dir outputs/lora/qwen25vl3b_qkvo_llm_v2 \
  --target-mode qkvo \
  --epochs 2 \
  --batch-size 1 \
  --gradient-accumulation-steps 8
```

关键参数：

| 参数 | 说明 |
|---|---|
| `--model-path` | 基础模型路径 |
| `--train-file` | 训练数据，默认 `data/multimodal_eval/train_lora.jsonl` |
| `--val-file` | 验证数据，默认 `data/multimodal_eval/val_lora.jsonl` |
| `--output-dir` | LoRA 输出目录 |
| `--epochs` | 训练轮数 |
| `--learning-rate` | 学习率 |
| `--batch-size` | 单设备 batch size |
| `--gradient-accumulation-steps` | 梯度累积步数 |
| `--max-length` | 最大文本长度 |
| `--max-pixels` | 图片输入分辨率上限 |
| `--lora-r` | LoRA 秩 |
| `--lora-alpha` | LoRA alpha |
| `--lora-dropout` | LoRA dropout |
| `--target-mode` | LoRA 注入模块范围，支持 `qv`、`qkvo`、`qkvo_ffn`、`custom` |
| `--save-steps` | checkpoint 保存间隔 |
| `--seed` | 随机种子 |

三种常用消融设置：

| 设置 | target-mode | 含义 |
|---|---|---|
| QV | `qv` | 只微调 query/value 投影，参数最少 |
| QKVO | `qkvo` | 微调注意力投影层，平衡效果和成本 |
| QKVO+FFN | `qkvo_ffn` | 同时微调注意力和前馈网络投影，参数更多 |

### 11.3 `scripts/finetune/eval_qwen25vl_lora.py`

作用：单独评估 LoRA adapter 的生成效果。

### 12.1 `scripts/eval/eval_multimodal_chains.py`

```bash
  --stop-on-error
```

常用参数：

| 参数 | 说明 |
|---|---|

```text
本系统面向博物馆文物导览场景，主要解决观众不知道文物名称但希望通过图片提问的问题。

```

## 23. 常见问题排查

### 23.1 远程导览服务返回 500，提示 Ollama connection refused

原因：远程服务器上 Ollama 没启动，或端口不是 `11434`。

处理：

```bash
export OLLAMA_MODELS=/root/autodl-tmp/ollama_models
nohup ollama serve > ollama.log 2>&1 &
ollama list
```

### 23.2 提示 Hugging Face 无法连接或本地无缓存

原因：远程环境断网或设置了 `local_files_only=True`，但模型没有下载到本地。

处理：

| 检查项 | 说明 |
|---|---|
| Qwen2.5-VL 模型路径 | 确认 `/root/autodl-tmp/models/Qwen2.5-VL-3B-Instruct` 存在 |
| embedding 模型缓存 | 确认文本和图片向量模型已缓存 |
| config 配置 | 确认 `embedding.model_name` 和 `image_index.model_name` 指向可用模型 |

### 23.3 远程服务启动了，但本地网页连不上

检查：

| 检查项 | 命令或说明 |
|---|---|
| 远程服务是否启动 | `curl http://127.0.0.1:8000/health` |
| SSH 隧道是否建立 | 本地运行 `ssh -p 21870 -L 8000:127.0.0.1:8000 root@connect.cqa1.seetacloud.com` |
| 网页地址是否正确 | Streamlit 中填写 `http://127.0.0.1:8000` |

### 23.4 训练中 SSH 断开

如果没有使用 `tmux` 或 `screen`，训练进程可能已经停止。

检查：

```bash
ps -ef | grep train_qwen25vl_lora | grep -v grep
nvidia-smi
```

如果没有训练进程，需要从最近 checkpoint 重新启动训练或直接使用已保存 checkpoint。

### 23.5 中文显示乱码

如果 PowerShell 中直接 `Get-Content` 显示乱码，但 Python 按 UTF-8 读取正常，通常是终端编码显示问题。

检查文件是否真实乱码：

```bash
python - <<'PY'
from pathlib import Path
print(Path("README.md").read_text(encoding="utf-8")[:300])
PY
```

Windows PowerShell 可尝试：

```powershell
chcp 65001
```

### 23.6 图像检索错误

可能原因：

| 原因 | 说明 |
|---|---|
| 图片质量差 | 上传图片模糊、角度差异大 |
| 文物外观相似 | 器形、纹饰或颜色相近 |
| 索引图片不足 | 某些文物只有一张或少量图片 |
| 阈值设置不合适 | `min_score` 或 `min_gap` 太严格或太宽松 |

相关配置：

```yaml
image_index:
  top_k: 5
  min_score: 0.55
  min_gap: 0.03
```

## 27. 总结

本项目不是单一模型调用 demo，而是一个完整的文物导览问答系统原型。它包含：

| 能力 | 对应实现 |
|---|---|
| 文物知识组织 | `data/`、`scripts/build/prepare_combined_kb.py` |
| 文本检索 | `src/embedder.py`、`src/retriever.py` |
| 图片检索 | `src/image_embedder.py`、`src/image_retriever.py` |
| 文本 RAG | `src/prompt.py`、`src/llm.py` |
| 多模态问答 | `src/hf_qwen_vl.py` |
| LoRA 微调 | `scripts/finetune/` |
| 远程推理服务 | `scripts/serve/serve_qwen25vl_lora_api.py` |
| 网页展示 | `app.py` |
| 语音播报 | `src/tts.py` |
| 自动评测 | `scripts/eval/` |
| 导览质量裁判 | `scripts/judge/` |

答辩时可以强调三点：

| 重点 | 说明 |
|---|---|
| 系统完整 | 从数据、索引、模型、网页到评测都有实现 |
| 设计可解释 | 方案 A 和 B4 都保留候选文物和检索上下文 |
| 实验可比较 | 统一五链路评测支持 RAG、LoRA 和多模态能力的消融分析 |
